# 設計の発展方針 (Roadmap)

現行の**稼働版**は README に記載のとおり **PostgreSQL + FastAPI + React** の 3 層構成です。
以下は「すでに設計として方向づけているが、まだ実装していない」発展方針です。
現状と将来を区別して記録することで、設計判断の意図を残します。

## 1. SQLite ローカルレプリカによる真のオフラインファースト

**現状**：オフラインファースト＝「Windows PC 1台でオンプレ完結して動く（クラウド非依存）」という意味で達成済み。
DB は PostgreSQL 単一インスタンス。

**発展方針**：複数拠点（例：工場と事務所）で、ネットワーク断でも各拠点が読み書きを続けられるよう、
**拠点ローカルに SQLite レプリカ**を置き、オンライン復帰時に PostgreSQL へ同期するモデルを検討している。

- 読み取りは常にローカル SQLite（低遅延・断線耐性）
- 書き込みはローカルに記録し、同期キュー経由で PostgreSQL に反映
- 競合解決はドメイン特性を利用（在庫の加減算は可換な操作が多く、イベントソーシング的に積み上げ可能）

**設計上の論点**：トリガーで守っている整合性（負在庫防止など）を SQLite 側でどう担保するか、
同期時の二重計上をどう防ぐか。ここは PostgreSQL 一本の現行構成で「正しさ」を固めてから着手する。

## 2. サーバーサイド PDF 生成

**現状**：PDF は印刷最適化ページ（CSS `@page`）＋ブラウザ印刷で出力。
帳票の見た目を Excel 時代と一致させることを優先した結果、人手の印刷操作が前提になっている。

**発展方針**：定期帳票の無人生成（バッチ）や、メール添付向けに、
**WeasyPrint 等のサーバーサイド PDF エンジン**で同じ HTML/CSS をヘッドレス描画する経路を追加する。
印刷ページの CSS をそのまま再利用できる設計にしてある。
